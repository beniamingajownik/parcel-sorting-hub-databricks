## 1. Data Source (Python Script)

The underlying lifecycle data is generated dynamically using Python script (generated by AI). This localized execution engine simulates real-world distribution mechanics over a 20-day operational window starting May 1, 2026.

### How the Engine Works
The generation scripts mimic high-throughput hub activity by mapping real-world logistical dependencies:
* **Traceable Lineage:** A randomized package tracking engine links unique 24-digit parcel barcodes across complex inbound truck deliveries (`DLV`), sorting routes, and outbound loading sheets (`LOD`).
* **Service Tier Logic:** Volatile distribution profiles apply distinct behavioral weights to specific service tiers (`PRIORITY`, `COURIER_STD`, `LOCKER_24H`), governing processing speeds and automated staging goals.
* **Failure Simulation:** The environment injects an intentional error rate into automated sensor streams, reproducing realistic device anomalies (e.g., label scuffs, calibration drops, sorting tray jams).

### Generated Files
The scripts output structured CSV logs across two separate execution contexts:
* **Master Configurations:** Global entity tables generated once to serve as system lookups (`devices_master.csv`, `employees.csv`).
* **Daily Ingest Batches:** A 7-file package emitted for every simulated operational day, cleanly timestamped:
  * `_parcels.csv` – Spatial metrics, base weights, and routing hubs.
  * `_status_history.csv` – Sequential state changes tracking the shipment lifecycles.
  * `_scans.csv` – Automated scanner and dimensioner event checkpoints.
  * `_sorting.csv` – Physical channelling logs from automated sorter chutes.
  * `_exceptions.csv` – Flagged anomalies mapping faults to resolving employees.
  * `_intake.csv` – Linehaul truck arrival times and inbound dock allocation logs.
  * `_loading.csv` – Outbound trailer departures and staging metrics.

## 2. Data Storage (AWS S3)

The pipeline isolates compute nodes from storage by anchoring the primary landing zone inside a scalable AWS S3 bucket architecture.

* **Directory Partitioning Strategy:** Raw files stream continuously into structured, isolated S3 prefixes organized by core entity names. This hierarchy ensures tidy cloud storage, acts as a staging ground for multi-date pipelines, and prevents massive cross-directory lookups.
* **Object Immutability:** The raw landing directories are treated as a read-only write-once ledger, preserving raw tracking anomalies for engineering troubleshooting and forensic log re-indexing.

## 3. Data Ingestion (Databricks Ingest)

The transformation workflow begins by pulling text-heavy data directly from the S3 landing zones into Databricks file processing systems.

* **Incremental Accumulation:** Ingestion utilizes custom file-tracking mechanisms that scan directory files incrementally. The environment processes newly arrived file objects since the last structural runtime, bypassing older rows to eliminate redundant cluster consumption.
* **Lineage Appending:** Upon hitting the pipeline threshold, raw dataset attributes are stamped with critical execution trackers. The engine appends system metadata tracking the specific source file name and an immutable ingestion timestamp, creating a clear audit trail for debugging data drift.

```sql
COPY INTO parcel_sorting_hub.bronze.exceptions
FROM (
  SELECT 
    exception_id,
    parcel_id,   
    device_id,   
    error_code,  
    status,      
    reported_at, 
    resolved_at, 
    employee_id,
    _metadata.file_name AS _source_file,
    current_timestamp() AS _ingested_at
  FROM 's3://my-s3-bucket/parcel_sorting_hub/data-raw-daily-batch/exceptions/'
)
FILEFORMAT = CSV
FORMAT_OPTIONS ('header' = 'true', 'inferSchema' = 'false');
```

## 4. Medallion Architecture & Data Pipeline

Data processes through a rigorous structural refinement layout, systematically scrubbing out errors and structuring models for reporting.

### Bronze Layer: The Raw Append Log
The incoming file payload streams directly into raw Bronze Delta tables. This layer enforces no strict data type validation or string formatting, acting as an immutable historical record of the event stream.

### Silver Layer: Relational Integrity & Cleaning
To safely build data structures without causing dropped relationships, the processing jobs enforce a strict **3-Tier Dependency Hierarchy**:
1. **Tier 1 (Parents):** Foundational nodes containing independent attributes (`devices`, `employees`, `intake`, `loading`) validate first.
2. **Tier 2 (Core Entities):** The main transactional entity (`parcels`) is processed next, cross-validating its relational keys directly against Tier 1 tables.
3. **Tier 3 (Child Events):** High-frequency situational log tables (`scans`, `exceptions`, `sorting`, `status_history`) validate last, matching against Tier 2 records.

#### The Quarantine Exception Pattern
Rather than allowing schema drift or empty structural keys to break processing runs, data goes through conditional evaluation checks inside temporary staging views:
* **The Routing Filter:** Fields are systematically parsed for empty string values, structural key formats, invalid weights, and corrupt timestamps.
```
    -- E X A M P L E
        WHEN zone IS NULL 
            OR TRIM(zone) = ''           
        THEN 'NULL_OR_EMPTY_ZONE'

        WHEN LOWER(TRIM(zone)) = 'null'
        THEN 'STRING_NULL_IN_ZONE'

        WHEN TRIM(zone) NOT IN (
            'Sort_Area', 
            'Inbound', 
            'Outbound'
        )                          
        THEN 'INVALID_ZONE'
```
* **Quarantine Isolation:** Records failing any check are flagged with a specific error string and routed to a dedicated quarantine table. Production transformations remain clean, while bad data stays accessible for developer auditing.
```
    -- E X A M P L E
      INSERT INTO parcel_sorting_hub.quarantine.tableX
      SELECT 
          ..._id,        
          ...type,            
          _source_file,                    
          _rejection_reason,
          current_timestamp() AS _rejected_at
      FROM validated_devices
      WHERE _rejection_reason IS NOT NULL
```

* **Windowed Deduplication:** Valid records inside the views go through deep window functions, stripping away matching duplicates from overlapping batch streams before merging securely into production Silver Delta tables.
```
    -- E X A M P L E
      MERGE INTO parcel_sorting_hub.silver.tableX AS target
      USING (
          WITH deduped AS (
              SELECT 
                  *,
                  ROW_NUMBER() OVER (PARTITION BY ..._id ORDER BY _ingested_at DESC) AS row_num_id,
                  ROW_NUMBER() OVER (PARTITION BY 
                      ..._id,
                      ...type,
                  ORDER BY _ingested_at DESC) row_dup
              FROM validated_devices
              WHERE _rejection_reason IS NULL
              )
          
          SELECT
              TRIM(..._id)        AS ..._id,
              TRIM(...type)       AS ...type,
              current_timestamp() AS _updated_at
          FROM deduped
          WHERE 
              row_num_id = 1 
          AND 
              row_dup = 1

      ) AS source
      ON target.device_id = source.device_id
      WHEN MATCHED THEN
          UPDATE SET *

      WHEN NOT MATCHED THEN
          INSERT *
```

## 5. Power BI Connection & Model Layer

Refined Silver layer tables are transformed into Gold tables using a multi-dimensional star schema, serving as the semantic reporting platform for Power BI visualizations.

### The Semantic Gold Model
* **Fact Tables:** Dense dimensional tables contain measurable supply chain attributes (`fact_parcel_journey`, `fact_scan_events`, `fact_exceptions`), storing transit durations, terminal errors, and custom metrics.
* **Dimension Lookups:** Conformed time structures (`dim_date`, `dim_hour`) group hours into static corporate work shifts (`MORNING`, `AFTERNOON`, `NIGHT`), taking all temporal logic burdens off the dashboard rendering engine.

### Power BI Engine Design
* **Hybrid Storage Architecture:** High-volume transaction facts use **DirectQuery Mode** for near real-time visibility into the sorting floors. Static reference tables use **Import Mode** to cache lookup variables in memory, providing fast interactive slicing.
